## 1. 뉴스가 끝난 뒤, 기대는 어떻게 달라졌을까? — 20초

안녕하세요. 저희는 2팀 손명관, 강도원, 이종혁, 김도언입니다.

뉴스는 무슨 일이 있었는지는 빠르게 알려줍니다. 하지만 그 이후 공개 데이터에 반영된 기대가 언제, 얼마나 달라졌는지는 한눈에 보기 어렵습니다.

Outlook AI Signals는 공개 예측시장 데이터를 시간의 흐름으로 정리해, 글로벌 이슈에 반영된 기대 변화를 읽도록 돕는 모니터링 대시보드입니다.

**전환:** 먼저 저희가 해결하려 한 문제부터 말씀드리겠습니다.

## 2. 이슈는 넘치지만, 변화의 강도는 한눈에 보이지 않습니다 — 25초

글로벌 이슈를 이해할 때 우리는 뉴스, 소셜 피드, 전문 보고서를 함께 봅니다.

뉴스는 사건을 빠르게 전하지만 기대의 변화량까지 연결해 주지는 않습니다. 소셜 피드는 반응은 빠르지만 여러 관점과 잡음이 섞입니다. 전문 보고서는 깊이가 있지만 매일의 변화를 따라가기에는 시간이 필요합니다.

결국 사용자가 세 정보를 직접 연결해 무엇이, 언제, 얼마나 달라졌는지를 판단해야 합니다. 저희는 이 연결에 드는 시간을 줄이고자 했습니다.

**전환:** 그 실마리를 공개 데이터에 남은 시간 기록에서 찾았습니다.

## 3. 공개 데이터에는 기대 변화가 남습니다 — 25초

정책 발표, 공식 일정 변경, 새로운 조사 결과처럼 사건 중심 정보가 나온 뒤에는 공개 예측시장 데이터에도 참여자 기대가 움직인 기록이 남습니다.

저희는 현재 값 하나만 보여주지 않습니다. 화면 예시처럼 24시간에 플러스 8퍼센트포인트, 7일에 플러스 13퍼센트포인트와 같이 변화의 방향과 크기를 함께 보여주고, 어느 구간에서 변화가 커졌는지도 시간축에서 확인하게 합니다.

다만 이 값은 전체 대중의 의견이나 실제 결과의 사실 확률이 아닙니다. 특정 공개 집계 데이터에서 관측된 기대값이라는 해석 범위를 항상 같이 표시합니다.

**전환:** 이 아이디어를 5일 동안 실제 서비스 흐름으로 완성했습니다.

## 4. 5일 동안 핵심 흐름을 완성했습니다 — 20초

첫날에는 제품 범위와 데이터 구조를 검증했습니다. 둘째 날에는 수집 경로와 홈 화면을 만들었고, 셋째 날에는 상세 화면과 차트, 해석 주의를 연결했습니다.

넷째 날에는 시나리오 대화 챗봇, 근거가 제한된 AI 브리핑과 안전 검증을 구현했고, 마지막 날에는 통합 테스트와 시연 준비, 배포작업까지 마쳤습니다.

짧은 일정에서 기능 수를 늘리기보다, 이슈 발견부터 신중한 해석까지 한 흐름이 끊기지 않도록 만드는 데 집중했습니다.

**전환:** 이 과정을 네 명이 역할을 나눠 진행했습니다.

## 5. 4명이 역할을 나눴습니다 — 25초

저는 PM과 기획을 맡아 전체 일정과 범위를 조율하고, 전 영역의 구현 지원, 문구와 안전 검토, 통합 리뷰와 발표 구성을 담당했습니다.

강도원 팀원은 홈과 상세 화면, 차트, 반응형 UI와 접근성을 구현했습니다. 이종혁 팀원은 데이터베이스와 API, 수집 및 복구 경로를 맡았습니다. 김도언 팀원은 데이터 정규화와 변화 지표, 근거 검증과 브리핑 생성을 담당했습니다.

역할은 나눴지만 인터페이스와 실패 상태는 함께 점검해, 각 영역이 하나의 제품 경험으로 이어지게 했습니다.

**전환:** 다음은 실제 구현에 사용한 기술입니다.

## 6. 화면·데이터·AI를 이 기술 스택으로 구현했습니다 — 25초

프런트엔드는 React 18, TypeScript, Vite, Tailwind CSS, Recharts로 반응형 화면과 선형 차트를 구성했습니다.

백엔드는 Python 3.11과 FastAPI로 REST API, SSE, 배치 수집, 독립 작업자를 구현했습니다. 데이터는 PostgreSQL에 누적하고, Polymarket의 Gamma와 CLOB 공개 데이터를 정규화해 24시간과 7일 변화를 계산합니다.

AI 생성은 OpenRouter 호환 경로를 사용하며, GitHub Actions, Docker Compose, Caddy로 수집과 실행 환경을 구성했습니다. 화면에는 검증을 통과한 결과만 전달됩니다.

**전환:** 여기서 중요한 것은 기술의 개수보다 경로를 어떻게 분리했는가입니다.

## 7. 수집·생성 경로를 분리했습니다 — 25초

일반 조회 경로는 공개 API에서 데이터를 받아 4시간 주기의 Python 수집기로 정규화하고, PostgreSQL에 누적한 뒤 FastAPI와 React로 전달합니다.

AI 생성은 이 경로에 섞지 않았습니다. 사용자가 브리핑을 요청했을 때만 독립 작업자가 저장된 근거를 읽고 결과를 만듭니다. API는 요청 상태만 기록하고, 생성 제공자에 직접 연결하지 않습니다.

이 분리 덕분에 정기 수집이나 일반 화면 조회는 생성 지연의 영향을 덜 받고, 생성이 실패해도 이전의 정상 브리핑을 유지할 수 있습니다.

**전환:** 이제 사용자가 실제로 보는 화면을 설명드리겠습니다.

## 8. 홈은 오늘 가장 크게 변한 이슈부터 보여줍니다 — 25초

홈 화면의 목적은 사용자가 먼저 살펴볼 이슈를 빠르게 찾게 하는 것입니다.

각 이슈에는 현재 반영 기대값과 24시간 또는 7일 변화가 표시됩니다. 동시에 데이터가 충분한지, 해석에 주의가 필요한지, 데이터가 제한적인지도 배지로 보여줍니다.

그리고 모든 수치와 같은 화면에 데이터 기준 시각을 둡니다. 따라서 사용자는 변화가 큰 이슈를 찾으면서도 그 숫자를 어디까지 읽을 수 있는지 함께 확인합니다.

**전환:** 이슈를 선택하면 단일 값보다 중요한 시간의 흐름으로 이동합니다.

## 9. 상세 화면은 변화의 흐름을 보여줍니다 — 30초

상세 화면에서는 이슈 정의, 현재 반영 기대값, 24시간과 7일 변화, 기간별 선형 차트를 한곳에서 확인합니다.

화면 예시의 96퍼센트와 7일 플러스 14퍼센트포인트는 실제 결과를 단정하는 숫자가 아니라, 해당 시각의 공개 데이터에 반영된 값과 관측 변화입니다.

차트에서는 큰 변화 구간을 표시하고, 공개 자료와 출처, 해석 주의, 데이터 기준 시각을 함께 제공합니다. 핵심은 숫자를 크게 보여주는 것이 아니라 언제, 어느 정도 달라졌는지를 근거와 한계까지 포함해 읽게 하는 것입니다.

**전환:** 이 화면을 이용하는 전체 흐름은 네 단계입니다.

## 10. 10초 안에 변화를 발견하고 확인합니다 — 20초

첫째, 24시간과 7일 변화가 큰 이슈를 발견합니다.

둘째, 상세 차트와 큰 변화 구간으로 방향과 크기를 확인합니다.

셋째, 브리핑과 공개 출처로 현재 확인 가능한 내용을 점검합니다.

마지막으로 기준 시각과 해석 주의를 보며 데이터가 말할 수 있는 범위를 구분합니다.

**전환:** 이 가운데 브리핑은 생성보다 검증을 먼저 설계했습니다.

## 11. AI 브리핑은 검증 후 공개됩니다 — 30초

브리핑은 자유롭게 작성된 문장을 바로 화면에 보내지 않습니다.

먼저 이슈 정의, 관측 수치, 허용된 공개 출처를 고정합니다. 그 근거 묶음으로 한 번의 생성 요청을 수행한 뒤, 완성된 문단 단위로 수치, 출처, 표현, 순서를 다시 검사합니다.

검증을 마친 완성 블록만 순서대로 화면에 표시되고, 작성 중인 조각은 공개되지 않습니다. 현재 시도가 실패하면 부분 결과를 제거하고 이전 정상 브리핑을 유지합니다.

**전환:** 그리고 현재 정보와 사용자의 가정도 서로 다른 공간에 두었습니다.

## 12. 확인된 브리핑과 조건부 대화를 분리했습니다 — 30초

왼쪽의 AI 이슈 브리핑은 이슈 정의, 관측 수치, 허용된 공개 자료, 기준 시각과 주의사항을 안정적으로 읽는 핵심 경험입니다.

오른쪽의 시나리오 대화는 사용자가 제시한 가정을 조건부로 살펴보는 별도 경험입니다. 하나의 이슈에 묶인 익명 세션으로 동작하고, 24시간과 최대 8개 사용자 턴으로 제한됩니다. 외부 도구에는 접근하지 않으며 검증된 완성 문단만 순서대로 표시합니다.

핵심은 가정이 현재 정보처럼 섞이지 않도록 구조적으로 구분한 것입니다. 애플리케이션 기본 설정에서는 비활성화되고, 운영 구성에서만 명시적으로 활성화됩니다.

**전환:** 이런 경계는 안내 문구만이 아니라 제품 구조에 들어가 있습니다.

## 13. 안전장치는 제품 구조로 구현했습니다 — 25초

첫째, 모든 수치에 데이터 기준 시각과 해석 주의를 함께 표시합니다.

둘째, 생성된 결과는 수치, 출처, 표현을 같은 규칙으로 다시 검증합니다. 셋째, AI가 사용할 수 있는 근거를 사전에 고정합니다. 넷째, 개인 단위 탐색 없이 이슈 단위의 공개 집계 데이터만 제공합니다.

그래서 실제 결과를 단정하지 않고, 공개 자료와 관측 변화 사이에 임의의 관계를 붙이지 않으며, 개인을 식별하는 화면도 만들지 않습니다.

**전환:** 지금까지 설명한 흐름을 실제 시연 영상으로 확인하겠습니다.

## 15. 실패·복구 방식까지 검증했습니다 — 25초

저희는 정상 화면만 확인하지 않았습니다. 브리핑 생성 실패, 연결 중단, 데이터 부족 상태에서도 사용자가 잘못된 내용을 보지 않고 안전하게 복구되는지를 점검했습니다.

백엔드는 Ruff 검사와 전체 550개 회귀 테스트를 통과했습니다. 프런트엔드는 320픽셀의 좁은 화면까지 확인했고, 포맷, 타입, 린트, 빌드 검사를 통과했습니다.

또한 SSE 재생, 연결 중단 시 폴링 대체, 이전 정상 결과 유지, 잘못된 수치와 표현 차단까지 검증했습니다.

**전환:** 이 과정에서 얻은 가장 큰 배움은 기능의 수보다 경계가 중요하다는 점이었습니다.

## 16. 기능보다 경계를 설계하는 과정이 중요했습니다 — 25초

첫째, 데이터는 숫자만 보여줄 때보다 기준 시각과 해석 범위를 함께 보여줄 때 의미가 생겼습니다.

둘째, AI는 많이 작성하는 것보다 검증된 문단만 공개하는 구조가 중요했습니다.

셋째, 역할을 나누더라도 정상 흐름뿐 아니라 실패와 복구까지 함께 점검해야 제품의 완성도가 높아졌습니다.

저희는 빠르게 만드는 데서 멈추지 않고, 안전하게 설명하고 다시 확인할 수 있는 제품을 만드는 데 집중했습니다.

**전환:** 이제 처음에 드린 질문으로 돌아가겠습니다.

## 17. 기대가 변한 순간을 읽는 도구입니다 — 20초

Outlook AI Signals는 미래를 대신 판단하는 화면이 아닙니다.

## 발표자용 한 장 큐시트

1. **문제 제기** — 뉴스 이후 기대 변화의 크기와 시점은 바로 보이지 않는다.
2. **기존 정보의 한계** — 뉴스·소셜·보고서를 사용자가 직접 연결한다.
3. **데이터 차별점** — 현재 값 하나가 아니라 24시간·7일의 관측 흐름을 읽는다.
4. **5일 일정** — 범위·데이터 → 수집·홈 → 상세·차트 → 브리핑·검증 → 통합 QA.
5. **팀** — 손명관 PM, 강도원 프런트엔드, 이종혁 백엔드, 김도언 데이터·AI.
6. **기술** — React/FastAPI/PostgreSQL 중심, 선형 차트와 독립 작업자.
7. **구조** — 4시간 수집 경로와 사용자 요청 기반 생성 경로를 분리한다.
8. **홈** — 값·변화·주의 상태·기준 시각을 함께 표시한다.
9. **상세** — 숫자보다 시간 흐름과 큰 변화 구간을 보여준다.
10. **사용 흐름** — 발견 → 변화 확인 → 근거 점검 → 신중한 해석.
11. **브리핑** — 고정 근거 → 작성 → 문단 검증 → 완성 블록 공개.
12. **경험 분리** — 현재 브리핑과 조건부 가정을 구조적으로 나눈다.
13. **안전 구조** — 화면 표시·출력 검증·근거 제한·집계 데이터.
14. **시연** — 홈 → 상세 차트 → 브리핑 → 해석 안내 → 별도 시나리오 대화.
15. **검증** — 550개 백엔드 테스트, 320픽셀 UI, 프런트엔드 주요 검사.
16. **배움** — 숫자에는 맥락, AI에는 검증, 통합에는 실패·복구 점검이 필요하다.
17. **마무리** — 미래 판단이 아니라 기대 변화의 흐름을 읽는 도구.
18. **질문** — 핵심 경계를 유지하며 간결하게 답한다.
