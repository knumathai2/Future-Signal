# TASK-055 — 맥락 후보와 AI 요약 연결 전략 분석

_작성일: 2026-07-10_

_브랜치: `pm/TASK-055-context-summary-strategy`_

_범위: 분석과 문서화만 수행. 제품 코드, DB, API, 정책은 변경하지 않음._

## 1. 결론

현재 문제는 안전 제약이 많아서라기보다, 제약의 대상을 **주장**이 아닌
**맥락 입력 자체**로 잡은 데서 발생한다.

프로젝트 문서가 막으려는 것은 다음 세 가지다.

1. 외부 사건을 변화의 원인처럼 단정하는 것
2. 검토되지 않은 외부 정보를 자동으로 공개하는 것
3. AI가 구조화된 근거 밖의 해석을 새로 만드는 것

반대로 문서가 맥락 자체를 금지하는 것은 아니다. PRD와 Service Design은
대표 이슈 3~5개에 수동 검토한 맥락 후보를 붙이고, 변화와 함께 확인할
정보로 제공하는 흐름을 명시한다. 따라서 적절한 방향은 제한을 없애는 것이
아니라 다음처럼 경계를 다시 긋는 것이다.

> **사람이 검토한 근거 묶음이 변화와 맥락을 연결하고, AI는 그 묶음 안에서만
> 문장을 정리한다. 후보 탐색, 관계 판단, 사실 추가는 AI가 맡지 않는다.**

제품 단위도 “시장 하나의 일반 설명”보다 **변화 에피소드**가 되어야 한다.
한 에피소드는 관찰된 변화, 같은 구간에서 확인할 맥락 후보, 추가 확인 사항,
해석 한계를 한 덩어리로 보여 준다.

### 후속 목표 갱신

이 최초 분석은 현재 프로젝트 정책 안에서 사람 검토를 유지하는 가장 안전한
경로를 제안했다. 이후 사용자는 사람 검토를 제거하고 기존 OpenRouter 경로를
사용하는 자동 수집·검증 구조를 목표로 정했다. 후속 실행 계획은 사람 검토를
다음 세 가지로 대체하되, 아직 승인되지 않은 정책 변경으로 취급한다.

1. URL·날짜·도메인·엔티티의 결정적 hard gate
2. 연구 모델과 다른 독립 검증 모델의 일치
3. 조건 미달 후보를 사용자에게 표시하지 않는 fail-closed 동작

구현 기준은 `reports/task-055-automated-context-execution-plan.md`와
`reports/task-055-automated-context-stretch-plan.md`이며, 이 문서의 사람 검토
흐름은 현재 정책 안에서 가능한 fallback 분석으로 남긴다.

## 2. 현재 상태에서 확인된 단절

### 2.1 수집 경로가 정기 데이터 흐름과 분리되어 있다

- `related_events`는 정기 수집 대상이 아니다. 별도 로컬/개발용 seed에
  고정된 4개 이슈만 들어 있다.
- 해당 seed는 운영상 자동 실행되지 않으며, 현재 로컬 동기화 기록에는
  67개 시장과 115개 리포트가 있지만 `related_events`는 0개로 남아 있다.
- 로컬 API의 상위 5개 이슈를 읽기 전용으로 확인한 결과, 모두
  `related_events=[]`였고 저장된 v3 리포트는 `external_context=null` 및
  동일한 후보 없음 문장을 반환했다.
- AI 리포트는 heat 순 상위 10개를 중심으로 생성되지만, 고정 seed의 4개
  이슈가 그 대상과 일치한다는 보장이 없다. 후보를 준비해도 실제 사용자가
  여는 리포트에 붙지 않을 수 있다.

즉, “수동 검토만 허용”이라는 정책은 존재하지만, **발견 → 검토 → 반영 →
리포트 재생성 → 화면 확인**의 운영 절차가 없다.

### 2.2 AI와 맥락 후보가 의도적으로 서로를 보지 못한다

`ReportPromptInputs`에는 후보 제목·날짜·메모가 있지만, 현재 `build_prompt()`는
이를 모델 입력에서 제외한다. AI가 작성하는 것은 아래 세 필드뿐이다.

- `issue_overview`
- `current_data_reading`
- `possible_outlook`

후보 관련 `possible_drivers`와 `external_context`는 별도 고정 조립 경로로
붙는다. 이 선택은 안전성에는 유리하지만 다음 부작용을 만든다.

- AI가 “무엇이 바뀌었고 어떤 맥락을 함께 볼 수 있는지”를 한 흐름으로
  정리할 수 없다.
- 후보가 없으면 여러 이슈에서 동일한 부재 문장이 반복된다.
- 후보가 있어도 AI 요약과 맥락 메모의 문체·언어·초점이 달라질 수 있다.
  현재 seed 메모는 영어이고 리포트 본문은 한국어다.
- 사용자는 `current_data_reading`, `external_context`, `possible_drivers`를
  각각 읽은 뒤 관계를 스스로 재조립해야 한다.

### 2.3 첫 번째 후보 하나만 사용하며 변화 시점과의 연결 규칙이 없다

현재 로더는 정렬 조건 없이 `limit(1)`로 후보 하나를 가져온다. 따라서 후보가
여러 개여도 다음을 보장하지 못한다.

- 변화 시점에 가장 가까운 후보인지
- 현재 선택 구간과 맞는 후보인지
- 검토자가 대표 후보로 지정한 항목인지
- 사용자가 볼 가치가 가장 높은 후보인지

현재 문구는 “기록 날짜와 움직임을 비교할 수 있다”고 말하지만, 그 비교 대상을
고르는 규칙은 없다.

### 2.4 리포트 구조가 관계보다 섹션 완결성에 최적화되어 있다

v3는 7~8개 섹션을 한 번에 하나씩 보여 준다. 구조 검증에는 좋지만 사용자가
원하는 핵심 흐름이 페이지 이동으로 분절된다.

1. 현재 데이터 읽기
2. 외부 맥락
3. 변화와 함께 확인할 요인
4. 조건부 전개
5. 추가 확인 사항
6. 데이터 한계
7. 해석 주의

특히 외부 맥락과 후보 비교가 서로 다른 섹션이고, 짧은 주의 문구와
`data_limitations`, `caution_note`가 겹쳐 실질 정보보다 방어 문구의 비중이
커진다.

### 2.5 출처와 검토 상태가 제품 데이터로 남지 않는다

현재 후보 데이터에는 제목, 날짜, 메모만 있다. 출처 URL, 출처 유형, 검토
시각, 검토 상태가 저장되거나 API로 제공되지 않는다. 따라서 사용자는
“독립적으로 확인해야 한다”는 안내를 보지만, 실제 확인을 시작할 근거는 화면에
없다. 이 상태에서는 맥락을 추가해도 신뢰와 활용성이 제한된다.

## 3. 제한의 문구보다 취지를 살리는 재해석

| 기존 제한 | 보호하려는 취지 | 과도하게 적용된 현재 방식 | 취지에 맞는 적용 방식 |
|---|---|---|---|
| 자동 외부 정보 매칭 금지 | 검토되지 않은 관계가 사실처럼 보이는 것을 방지 | 맥락 후보 수집 흐름 자체가 없음 | 비공개 탐색 보조 + 사람의 공개 승인 |
| 인과 단정 금지 | 시간상 인접한 사건이 변화의 이유처럼 보이는 것을 방지 | 후보를 AI 입력에서 완전히 제거 | 승인 후보와 관찰 변화의 시간 관계만 전달하고 관계 미입증 문구를 고정 |
| 고정 스키마 | 근거 밖의 서술과 형식 변동 방지 | 필드 수와 길이 중심의 제약 | 문장마다 근거 유형과 참조 ID를 제한 |
| 주의 문구 상시 노출 | 숫자의 과잉 해석 방지 | 여러 섹션에서 유사 문구 반복 | 짧은 상시 주의 + 상세 한계 한 곳으로 계층화 |
| 대표 이슈 3~5개만 수동 관리 | 해커톤 범위와 검토 품질 유지 | 코드에 seed가 있지만 실제 데이터 흐름과 분리 | 대표 이슈를 먼저 정하고 후보 커버리지와 리포트 생성을 함께 운영 |

핵심은 **제약의 양이 아니라 검증 단위**다. 현재는 “이 문장이 금지어를
포함하는가”와 “필드 길이가 맞는가”가 주 검증 단위다. 앞으로는 아래를 함께
검증해야 한다.

- 이 문장은 어떤 구조화 수치나 승인 후보를 근거로 하는가
- 후보가 실제로 검토된 항목인가
- 문장이 관찰, 후보, 조건부 읽기, 한계 중 어떤 유형인가
- 후보와 변화의 관계를 단정하지 않았는가

## 4. 제안 모델 — 변화 에피소드와 근거 묶음

### 4.1 사용자에게 보여 줄 기본 단위

한 이슈의 모든 설명을 반복하는 대신, 주요 변화 구간마다 아래 묶음을 만든다.

```text
변화 에피소드
├─ 관찰: 언제, 어느 구간에서, 몇 퍼센트포인트 움직였는가
├─ 맥락 후보: 같은 구간에 사람이 검토한 공개 사실 0~3개
├─ 연결 경계: 시간상 함께 확인할 수 있으나 관계는 입증되지 않음
├─ 추가 확인: 공식 문서, 날짜, 후속 공개 데이터에서 확인할 항목
└─ 해석 한계: 활동량, 유동성, 이력, 대표성
```

이 단위라면 사용자는 30초 안에 “무엇이 바뀌었는가, 어느 맥락을 같이 볼 수
있는가, 무엇을 더 확인해야 하는가”를 파악할 수 있다.

### 4.2 근거 묶음

AI에 원문이나 검색 결과를 직접 주지 않고, 먼저 아래처럼 검토된 구조체를
만든다.

```json
{
  "episode": {
    "window": "24h",
    "start_at": "...",
    "end_at": "...",
    "change_pp": -6.5,
    "inflection_at": "...",
    "data_as_of": "..."
  },
  "issue": {
    "title": "...",
    "tracked_condition": "...",
    "end_date": "..."
  },
  "approved_contexts": [
    {
      "candidate_id": "...",
      "title": "...",
      "occurred_at": "...",
      "neutral_summary_ko": "...",
      "source_label": "...",
      "source_url": "...",
      "review_status": "approved"
    }
  ],
  "limitations": ["low_activity", "short_history"]
}
```

이 구조에서 후보의 포함 여부와 요약 문장은 사람의 검토 결과다. AI는 후보를
찾거나 관계를 판단하지 않는다.

### 4.3 AI의 역할

AI는 “분석가”가 아니라 **근거 묶음 렌더러**로 둔다.

- 구조화 수치와 승인 후보만 사용한다.
- 외부 사실을 새로 추가하지 않는다.
- 각 문장은 `observation`, `context_candidate`, `verification`,
  `limitation` 중 하나의 유형을 가진다.
- 수치 문장은 metric ID, 맥락 문장은 candidate ID를 참조한다.
- 후보와 변화 사이에는 시간상 비교 문법만 허용한다.
- 후보가 없으면 긴 부재 설명을 반복하지 않고 맥락 섹션 자체를 숨긴다.

권장 서술 문법은 다음과 같다.

```text
[시각/구간] 공개 데이터에서 [방향/폭] 변화가 관찰됐습니다.
같은 구간에 함께 확인할 맥락 후보로 [제목/날짜]가 등록되어 있습니다.
현재 데이터만으로 이 후보와 관찰된 변화 사이의 관계는 확인할 수 없습니다.
추가로 [문서/날짜/후속 데이터]를 확인하면 변화의 지속성과 맥락을 구분하는 데 도움이 됩니다.
```

이 방식은 인과를 말하지 않으면서도 사용자가 원하는 연결감을 제공한다.

## 5. 맥락 후보 운영 흐름

### 5.1 발견

- 대상은 전체 이슈가 아니라 먼저 정한 대표 이슈 3~5개다.
- 검색이나 외부 자료 탐색은 비공개 PM 보조 단계에서 수행한다.
- 자동 도구는 후보 목록 초안까지만 만들 수 있고 공개 DB에 쓰지 않는다.
- 후보는 이슈의 핵심 주체, 기준 조건, 기준일, 변곡 시각을 조합해 찾는다.

### 5.2 검토

후보가 공개되려면 최소한 아래 항목을 사람이 확인한다.

- 공개적으로 확인 가능한 사건인가
- 사건 날짜가 출처에서 확인되는가
- 제목과 한 줄 요약이 출처 범위를 넘지 않는가
- 해당 변화 구간과 함께 볼 이유를 시간상 설명할 수 있는가
- 관계 단정 없이도 사용자에게 추가 확인 가치를 주는가
- 출처, 검토자, 검토 시각이 남아 있는가

### 5.3 승인과 반영

- 상태를 `draft → approved/rejected → stale`로 관리한다.
- 공개 경로는 `approved`만 읽는다.
- 대표 후보 순서는 사람이 정한다. 현재 정책에서 후보 자동 순위화는 허용되지
  않으므로, 단순 시간 거리 정렬도 새 ADR 전에는 공개 선택 기준으로 사용하지
  않는다.
- 승인 후보가 추가되면 해당 이슈의 리포트를 재생성 대상으로 표시한다.
- 리포트가 생성된 뒤 수치 일치, candidate ID 참조, 출처 표시, 관계 미입증
  문구, 데이터 기준 시각을 함께 검수한다.

### 5.4 만료와 갱신

- 후보는 영구 설명이 아니라 특정 변화 에피소드에 연결한다.
- 새 변곡점이 생기면 기존 후보를 자동 재사용하지 않고 다시 검토한다.
- 사건 자체가 정정되거나 출처가 바뀌면 `stale`로 전환하고 기존 리포트를
  공개 경로에서 제외하거나 재생성한다.

## 6. 사용자 경험 구성

현재 7~8단계 섹션 탐색보다 아래 3단 구성이 사용 목적에 가깝다.

### 6.1 한눈에 보는 변화

- 현재 공개 데이터 읽기
- 24시간/7일 변화 폭
- 가장 큰 변화 시각
- 데이터 기준 시각

### 6.2 함께 볼 맥락

- 승인 후보 제목, 날짜, 한 줄 요약
- 출처 링크와 수동 검토 표시
- “시간상 함께 확인할 정보이며 관계는 입증되지 않음” 고정 안내
- 차트 변곡점과 후보 날짜를 같은 타임라인에서 시각적으로 연결

### 6.3 다음 확인

- 이후 공개 데이터에서 변화가 유지·완화되는지
- 이슈의 기준 조건과 기준일에 공식 변경이 있는지
- 후보 출처에 후속 정정이나 추가 발표가 있는지
- 짧은 상시 주의 문구와 펼쳐 볼 수 있는 상세 데이터 한계

이렇게 하면 AI 요약은 일반 설명문이 아니라 콘텐츠 작성자와 리서처가 바로
브리핑 초안의 뼈대로 활용할 수 있는 구조가 된다.

## 7. 단계별 적용 범위

### 단계 A — 현 정책·스키마 안에서 회복 가능

1. 실제 상위/대표 이슈 3~5개를 먼저 고른다.
2. 기존 `related_events` 필드에 한국어 검토 메모와 날짜를 넣는다.
3. seed 실행, 리포트 재생성, API/UI 확인을 하나의 데모 준비 체크리스트로
   묶는다.
4. 후보가 있는 대표 이슈가 리포트 생성 상위 10개에 포함되도록 수동 데모
   준비 순서를 정한다.
5. 대표 이슈 후보 커버리지를 배포 전 점검 항목으로 둔다.

이 단계는 새 자동 매칭이나 자유 서술을 추가하지 않는다. 다만 DB 쓰기는 기존
승인 절차를 따라야 한다.

### 단계 B — 가장 작은 제품 개선

1. 현재 8개 API 필드는 유지한다.
2. `current_data_reading`, `external_context`, `possible_drivers`를 한 화면의
   변화 에피소드 카드로 묶는다.
3. 후보 없음 고정 문장은 숨기고, 후보가 있을 때만 맥락 블록을 보여 준다.
4. 상단에는 짧은 주의 문구만 남기고 상세 한계는 마지막 한 곳에 모은다.
5. 후보를 첫 행 하나로 고르지 않고, PM이 지정한 대표 후보를 사용한다.

이 단계는 코드 작업이 필요하지만 DB나 공개 API 모양은 유지할 수 있다.
ADR-033의 표시 순서와 ADR-034의 조립 결정을 다시 검토해야 한다.

### 단계 C — 권장 목표 구조

1. 후보에 출처 URL, 검토 상태, 검토 시각, 연결할 변화 에피소드 ID를 둔다.
2. 승인 후보로 근거 묶음을 만든다.
3. AI가 근거 참조가 포함된 짧은 연결 문장을 작성한다.
4. 저장 전 수치 일치, candidate ID 존재, 출처 존재, 관계 미입증 문구를
   기계적으로 검증한다.
5. 화면에서 각 맥락 문장을 실제 후보와 출처로 추적할 수 있게 한다.

이 단계는 DB 스키마와 공개 API, AI 입력 경계에 영향을 주므로 별도 사용자
승인과 ADR이 필요하다.

## 8. 성공 기준

| 영역 | 점검 기준 |
|---|---|
| 후보 커버리지 | 대표 이슈 3~5개 각각 승인 후보 1개 이상 |
| 출처 추적 | 공개되는 후보 100%에 제목·날짜·출처·검토 상태 존재 |
| 근거 추적 | 맥락 문장 100%가 승인 candidate ID를 참조 |
| 수치 정합성 | 요약의 값·구간·시각이 구조화 입력과 일치 |
| 안전성 | 관계 단정, 실제 결과 단정, 행동 유도 문장 0건 |
| 사용자 이해 | 30초 안에 변화·맥락 후보·추가 확인 항목을 설명 가능 |
| 활용성 | 콘텐츠 작성자/리서처가 요약을 브리핑 초안의 뼈대로 평가 |
| 반복 억제 | 후보 없음 안내와 상세 주의 문구가 한 화면에서 중복되지 않음 |

## 9. 권장 우선순위

1. **수집 운영 복구**: 대표 이슈와 승인 후보를 실제 데이터 경로에 넣는다.
2. **변화 에피소드 UI**: 데이터 읽기와 맥락 후보를 한 카드/타임라인으로 묶는다.
3. **근거 묶음 계약**: 후보 ID와 수치 근거가 있는 AI 입력·출력 계약을 정한다.
4. **출처·검토 메타데이터**: 별도 승인 후 스키마/API에 반영한다.
5. **비공개 후보 발견 보조**: 공개 반영은 계속 사람의 승인 뒤에만 수행한다.

가장 먼저 해야 할 일은 AI 프롬프트를 넓히는 것이 아니다. **대표 이슈마다
신뢰할 수 있는 후보가 실제로 존재하고, 그 후보가 어떤 변화 에피소드에
연결되는지 운영상 보장하는 것**이다. 그 다음에 AI를 근거 묶음 렌더러로
좁혀 사용하면 현재 제한의 취지를 유지하면서도 훨씬 유용한 결과를 만들 수
있다.

## 10. 현재 분석의 승인 경계

이번 TASK에서는 어떤 변경도 승인하거나 구현하지 않았다. 후속 작업 시 아래
항목은 별도 확인이 필요하다.

- AI에 승인 후보를 입력으로 제공하도록 ADR-034를 바꾸는 일
- ADR-033의 화면 순서 또는 공개 리포트 구조를 바꾸는 일
- 출처 URL·검토 상태·에피소드 연결 필드를 위한 DB 스키마 변경
- 후보 메타데이터를 공개 API에 추가하는 일
- 외부 자료 수집을 정기 작업에 넣는 일
- 공유/운영 DB에 후보를 쓰거나 리포트를 재생성하는 일

반면 비공개 후보 발견 보조, 수동 검토 절차 설계, 기존 필드 안의 대표 이슈
커버리지 점검은 현재 제한의 취지 안에서 후속 분석이 가능하다.
