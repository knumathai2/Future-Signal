## 백업 상황별 멘트

### 백엔드 또는 API를 사용할 수 없는 경우

"현재 화면은 로컬 백업 데이터를 표시하고 있습니다. 이 경우에도 데이터 기준
시각을 그대로 보여주며, 최신 수집이 된 것처럼 말하지 않습니다. 데모의 핵심은
동일합니다. 홈에서 이슈를 고르고, 상세 차트와 요약 영역, 해석 주의 문구를
확인합니다."

### 저장된 리포트가 없는 경우

"이 이슈에는 아직 저장된 템플릿 요약이 없을 수 있습니다. 이 상태도 의도된
화면입니다. 이슈 정보와 차트는 계속 확인할 수 있고, 저장 요약이 준비되면 같은
위치에 고정된 섹션 구조로 표시됩니다."

### 차트 이력이 부족한 경우

"선택한 기간의 기준 데이터가 충분하지 않으면 변화값을 억지로 채우지 않습니다.
대신 이력 부족 상태를 보여주고, 해석에 제한이 있음을 같은 화면에서 알립니다."

### 관련 사건 후보가 없는 경우

"모든 이슈에 관련 사건 후보가 있는 것은 아닙니다. 후보가 없는 경우에는 빈
상태를 그대로 보여주며, 자동으로 외부 뉴스를 연결하지 않습니다. 데모 전체를
보여줄 때는 수동 후보가 등록된 대표 이슈를 선택합니다."

## 스크린샷 및 리허설 체크리스트

필수 스크린샷:

- 상위 이슈 카드, 데이터 기준 시각, 해석 주의 배지가 보이는 홈 대시보드.
- 클릭 전 대표 이슈 카드.
- 반영 기대값과 지표 카드가 보이는 상세 헤더.
- 7일 기간이 선택되어 있고 해석 주의 배지와 데이터 기준 시각이 보이는 차트
  섹션.
- 선택 이슈에 따라 차트 표시점 또는 표시점 없음 상태.
- 저장된 v2 내용이 있을 때의 이슈 요약 카드.
- 백업용 이슈 요약 빈 상태.
- 전체 정보 안내 화면.
- 큐레이션된 이슈 하나의 관련 사건 후보 섹션.
- 라이브/로컬 API를 사용할 수 없을 때의 정적 백업 안내 또는 오래된 데이터
  상태.

5일차 리허설 전 필수 로컬/라이브 점검:

- 백엔드와 프론트엔드를 로컬에서 시작한다.
- 홈 -> 상세 이동이 작동하는지 확인한다.
- `/api/issues`가 라이브 데이터 또는 데이터 기준 시각이 명확한 백업 데이터를
  반환하는지 확인한다.
- `/api/issues/{id}/history?window=7d`가 차트 포인트 또는 이력 부족 상태를
  반환하는지 확인한다.
- `/api/issues/{id}/report`가 선택한 데모 이슈의 성공 응답 또는 중립적 빈
  상태를 반환하는지 확인한다.
- 선택한 데모 이슈에 관련 사건 후보가 있는지 확인하거나, 후보 없음 멘트를
  준비한다.
- 데스크톱과 모바일 너비에서 제목, 지표, 차트, 리포트 텍스트가 넘치지 않는지
  확인한다.

5일차 리허설 항목:

- 기본 데모 이슈 1개와 백업 이슈 1개를 고른다.
- 로컬/라이브 준비 상태를 확인한 뒤 스크린샷을 캡처한다.
- 4분 버전과 2분 압축 버전을 각각 연습한다.
- 심사 중 로컬 서버를 보여줄 수 없는 상황에 대비해 정적 스크린샷 순서를
  준비한다.
- UI 문구, 리포트 템플릿, 관련 사건 후보, 이 발표/데모 문구에 대해
  `TASK-018` 최종 문구 점검을 실행한다.

| 원인을 단정하는 문제는 어떻게 피하나요? | 뉴스를 움직임에 자동 연결하지 않습니다. 관련 사건 항목은 수동 큐레이션된 맥락 후보이며, 원인이 아니라 함께 확인할 자료로 표시합니다. |
| 왜 고정 섹션 요약을 쓰나요? | 요약을 미리 정한 섹션 안에 제한해 이슈 의미, 현재 데이터 흐름, 조건부 전개, 확인 지점, 해석 주의를 설명합니다. 자유 형식 해설로 넓어지지 않게 하기 위해서입니다. |
| 라이브 데이터를 사용할 수 없으면 어떻게 하나요? | API는 데이터 기준 시각이 명확한 정적 백업 데이터를 제공할 수 있습니다. 데모는 최신 수집처럼 말하지 않고도 같은 흐름을 보여줄 수 있습니다. |
| 5일 동안 실제로 만든 것은 무엇인가요? | React 대시보드/상세 흐름, FastAPI 읽기 엔드포인트, 정규화된 이슈 데이터, 지표 계산, 5pp 기준선 통과 표시, 해석 주의 배지, 저장 리포트 읽기 경로, v2 리포트 구조, 수동 큐레이션 맥락 후보를 만들었습니다. |
| 왜 MVP 범위가 이렇게 좁나요? | 위험은 개발 시간만이 아니라 제품 프레임에도 있습니다. MVP는 하나의 명확한 사용자 흐름에 집중하고, 계정, 저장 목록, 정기 브리핑, 자동 사건 연결은 해커톤 범위 밖에 둡니다. |
| MVP 이후에는 무엇을 확장할 수 있나요? | MVP 이후에는 콘텐츠용 내보내기, 저장된 이슈 검토, 주기적 브리핑, 더 넓은 카테고리, 더 강한 평가 체계를 검토할 수 있습니다. 단, 안전한 제품 프레임이 유지된 뒤에 확장합니다. |
